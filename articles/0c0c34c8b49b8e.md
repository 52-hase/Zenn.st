# 生成AIとは
- `生成AI`は、膨大なデータから学習し、`テキスト・画像・音声・動画`などの新しいコンテンツを自動的に創出する技術です。
- 単なる既存データの再現ではなく、指示に基づいた`独自の成果物`を生み出せます。


<br>


# 生成AIの概要
#### [ コード作成とテストの自動化 ]
- `初期コードの生成`、`テストケースの作成`、`仕様書のドラフト`など、反復的な作業を自動化し、開発効率を向上させます。

#### [ リアルタイムなコード補完と提案 ]
- `GitHub Copilot`などのツールは、開発中のコードに対して`リアルタイムで適切な補完や提案`を行い、開発者の生産性を高めます。

#### [ コードレビューとリファクタリングの支援 ]
- チャット機能を活用して、`コードの読み取り`、`レビュー`、`リファクタリングの提案`を行い、コードの品質向上を支援します。


<br>


# LLM（大規模言語モデル）とは？
#### [ 膨大なデータとパラメータによる自然な文章生成 ]
- `LLM`は、大量のテキストデータと数十億から数兆のパラメータを用いて学習し、`人間のような自然な文章`を生成できます。

#### [ 多様な言語処理タスクへの対応 ]
- `文章作成`、`翻訳`、`要約`など、さまざまな自然言語処理タスクに特化しており、従来の小規模モデルよりも正確で柔軟な応答が可能です。

#### [ 生成AIの中での位置づけ ]
- `LLM`は、生成AIの一分野であり、特に「`言語`」に関する生成に特化しています。生成AI全体では、`画像`・`音声`・`動画`なども含む幅広いコンテンツ作成技術を指します。


<br>

# AI駆動開発を行う上での注意点

#### [ ハルシネーションのリスク ]
- LLMが生成する架空の情報や誤ったコードに注意する
- 実在しない関数や動作しないプログラムが出力される可能性がある

#### [ リスクへの対策 ]
- 生成されたもので、自分が理解できないコードは使わない
- 内容を十分理解できないコードは採用しない
- 言語やフレームワークのバージョン違いによる問題も発生しうる
- 理解不足は余計な時間とエラーを生む
- 生成コードは参考・補助として扱う
- 必ず自身で動作確認と修正を行う
- 面接でも理解度がチェックされる時代
- 適切な知識レベルが必須
- 知識があれば開発効率は向上
- 知識不足は効率低下・品質低下・セキュリティリスクを招く
- 責任を負えない場合はAI使用を避けるべき

#### [ AI駆動開発の手順 ]
1. LLMでコードを生成
2. 生成されたコードを評価
3. 理解できないコードを特定
4. 理解できないコードを避ける
5. 生死絵されたコードをデバッグ
6. 必要な修正を加える


<br>


# トークン数に注意する

#### [ トークンとは ]
- `トークン`とは、LLMが文章を処理する`最小単位`のこと
- `単語`、`サブワード`、`記号`などに分解される
- 例「彼は手紙を書いています。」→「彼」「は」「手紙」「を」「書い」「て」「い」「ます」「。」
- トークンの連続から意味のある文章やコードが生成される

#### [ トークン制限と最適化 ]
- `LLM`の処理能力には上限がある
- 長すぎるコードは重要部分が切り捨てられる
- 不完全・誤ったコード生成のリスク増加


#### [ 活用方法 ]
- 重要な部分だけをピックアップして入力する
- コードを適切に`分割・整理`
- 必要な情報だけを効率よく伝える
- `正しい情報伝達`により、正確なコード生成とデバッグが可能になる


<br>


# プロンプト

#### [ プロンプトについて ]
- `プロンプト`とは、AIにやってほしいことを伝える「指示文」のこと
- 「`指示文`」は、タスクの説明、質問、システムの要件、コードの仕様など、具体的に書くほど、AIは実用的なコードや設計を出しやすくなる
- システム開発では以下を含めると効果的
・作りたいもの（要件）
・技術的な背景（使う言語や環境など）
・守るべきルール（制約条件）

#### [ プロンプトの役割について ]
- `プロンプトの質`が、AIの出力（コードなど）の質に直結する
- 曖昧な指示だと、誤ったコードが生成されることがある
- `明確かつ具体的な要件`を書くことが重要
- プロンプト次第で、生成されるコードの`正確性`、`効率性`、`可読性`に影響する
- セキュリティ基準やコーディング規約の遵守にも関わる


#### [ プロンプトの最適化戦略 ]
- 情報を整理する3つのポイント
###### ① 目的・意図
・何を実現したいか（機能や課題）を明確に伝える
例：「リストの重複を削除し、昇順ソートする関数」
###### ② コンテキスト（前提条件）
・実行環境やデータの前提を共有する
例：「Ruby 3.0以上、配列の要素は整数のみ」
###### ③ 制約条件・フォーマット
・コーディングルールや使用ライブラリ、処理方法の制限
例：「スネークケース、標準ライブラリのみ、例外処理あり」


<br>


# ロール設定
#### [ ロール設定について ]
- `ロール設定`とは、生成AIに特定の「役割」を与える方法のこと
- 現場で求められる品質や基準に近い、専門性と整合性のある成果物を得ることができる

#### [ ロール設定の目的と効果 ]

#### 　・ 専門性の向上（役割に応じて専門知識や最新手法が反映されやすい）
例：「セキュリティエンジニア」として、Webアプリの脆弱性を洗い出してください。
　→ 最新のセキュリティ知識に基づいた指摘が得られる。

#### 　・ 視点の一貫性の確保（事前設定により内容に統一感が生まれる）
例：「初学者向けの講師として、Reactの使い方を解説してください」
　→ 難しい用語を避けた丁寧な説明になり、初心者に優しい一貫した語り口になる。

#### 　・ 指示の明確化（どの分野・視点に注力するかを明示できる）
例：「Ruby on Rails を使って、ログイン機能を Devise で実装してください」
　→ 使用フレームワークとライブラリが明示されているため、出力内容がブレにくくなる。



#### [ 箇条書きによる条件指示 ]
- 箇条書きで条件を記載し、Rubyのメソッドを作成する。

```
『プロンプトの条件は、箇条書きで詳細かつ具体的に伝える』


・ 引数と返り値
メソッドは配列を引数として受け取り、配列を返す

・ 重複要素の削除
配列内の重複する要素をすべて削除する

・ 昇順ソート
重複が削除された配列を昇順に並べ替えて返す

・ メソッド名の命名規則
メソッド名はスネークケース（小文字とアンダースコア）で記述する

・ 使用するライブラリの制限
標準ライブラリのみを使用し、外部ライブラリは使用しない

・ エラーハンドリング
引数が配列でない場合や、配列内の要素が予期しない型である場合は、適切なエラーメッセージを返す
```


- 出力コード例
```ruby
def clean_and_sort(array)
  return 'Error: 配列を渡してください' unless array.is_a?(Array)

  array.uniq.sort
end
```

# プロンプトのスコープ
#### [ グローバルスコープ ]
- すべてのプロジェクトや作業に共通して適用されるルールや指示。
例：文体やスタイルガイドなど、一般的な指針を設定。

:::details グローバルスコープの例

- Ruby on Railsの開発者である場合、日常的に使用する技術スタックや開発スタイルに即した基本的なガイドラインをあらかじめ定義しておくことで、効率よくやり取りできます。ロール（役割）を指定するのも効果的です。
```
・ ロール固有のインストラクション
- 熟練したRailsエンジニアとして対応する
- 読みやすさとメンテナンス性の高いコードを優先する
-現在のRailsコミュニティで推奨されている実装方針を取り入れる
- 拡張性と高速なレスポンスを意識した設計を提案する
- セキュリティに配慮し、安全なコーディングを心がける
- 必要に応じてデザインパターンを導入し、構造的な解決策を提示する
- 冗長な処理や重複コードは整理し、改善案を提示する
- プロジェクト内でのコーディングスタイルの統一を意識する

・ 共通ルール
- Railsのコーディング規約と設計原則に準拠する
- Ruby 3系の最新構文を用いる
- RESTアーキテクチャを基本としたルーティングとリソース設計を行う
- コントローラーでのパラメータ処理には Strong Parameters を適用する
```
:::



#### [ プロジェクトスコープ ]
- 特定のプロジェクトやリポジトリに適用されるルールや指示。
例：技術スタックや命名規則など、プロジェクト固有の要件に合わせた詳細な設定。 

:::details プロジェクトスコープの例

- 特定の技術やこのプロジェクト固有のルールをここにまとめます。

```
・ コメントとドキュメント
- 複雑な処理にはコメントを添える
- クラスには目的を説明するコメントを書く
- 公開APIはYARD形式で記述
- TODOは明確かつ実行可能に
- コメントは簡潔な英語で


・ コーディング規約
- Ruby Style Guide に従う
- インデントは2スペース、1行120文字以内
- スネークケース（メソッド）、キャメルケース（クラス）、大文字スネーク（定数）を使う
- do...endと{}を使い分ける（1行は{}）
- ifを優先、ガード節で早期リターン
- メソッドは20行以内、クラスは100行以内を目安に


・ エラーハンドリング
- rescue_fromなどで例外を適切に処理
- カスタム例外クラスを活用
- ログ記録と通知を行う
- HTTPステータスやトランザクションも正しく使う


・ 変数名
- 意味のある名前にする
- ブール値はis_やhas_などを使う
- 配列は複数形にする
- 一時的な変数は短くてもOK（例: i, j）
- メソッド名は動詞で始める


・ MVCごとの方針
Models
- 関連・バリデーション・scopeを適切に定義

Views
- ERB + partialで整理、フォームヘルパーを活用

Controllers
- ロジックはモデルやサービスへ分離、before_actionで共通処理

テスト
- RSpecで書く
- FactoryBotでテストデータを用意する
```
:::



#### [ ユーザープロンプトスコープ ]
- 都度、特定のタスクや目的に応じて個別に指示を与えるプロンプト。
例：特定の関数の実装方法やバグ修正の手順など、個別の作業指示。

#### [ ラテラルシンキング ]
- 物事を論理的・直接的（ロジカル）に考えるのではなく、視点をずらして発想する思考法

::: details ラテラルシンキングの例
- 従来とは異なる角度からのアプローチや、分野を横断した視点を取り入れることで、思考の幅を広げる
```
・異分野との組み合わせを意識する（例：読書 × 音楽、料理 × プログラミング）
・目的ではなく手段に着目する（例：「集中するために読書」→「集中を高める方法全般に応用」）
・他人の立場で考えてみる（例：子ども、高齢者、外国人など）
・マイナスや不満を起点にする（例：「映画が長くて集中できない」→短編映像だけを推薦するアプリ）
・真逆の発想を試す（例：「時間を効率化」→「非効率を楽しむ」）
・過去・未来の視点を加える（例：昔のアニメと今の技術を融合）
・フィクションの世界で考える（例：もし自分がドラマの登場人物だったらどんなアプリを使う？）
```
:::

# 画面設計とUI/UXの効率化

#### [ UI/UX設計の効率化と高品質化 ]
- AIツールを活用することで、デザインのアイデア出しやプロトタイプ作成が迅速に行える
- ユーザー視点に基づいた高品質なUI/UX設計が実現できる。

#### [ 新しい発想や着想の支援 ]
- 多様なデザイン案やユーザーフィードバックの予測が可能
- AIによって人間では思いつかない視点やアイデアを得ることができる

#### [ 将来を見据えたAI活用スキルの重要性 ]
- コーディング作業の多くがAIに代替されると予想される中で、エンジニアはAIツールを効果的に使いこなすスキルを磨くことが求められている。


#### [主要UI生成AIツールと活用時の注意点 ]

- 現在の代表的なUI生成AIツールには、Uizard、v0、Visily、Relume などがある
- それぞれのツールには異なる特徴や得意分野があり、用途に応じて使い分けることが重要
- 全面的に依存するのではなく、補助的な役割として活用するのが望ましい
