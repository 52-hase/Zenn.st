```yaml

- README.md # すでにあるもの。documentディレクトリへのリンクを途中に入れる。例えば、「アーキテクチャはこちら」
- /document
  - hearing.md # 要件定義のドキュメント（スクールですでに書いたREADMEがあるので不要）
  - architecutre.md # アプリケーション全体のアーキテクチャ図、MVCに関する説明やMVCに沿った実装になっていることをアピール
  - implementation.md # リーダブルコードに書いてあるようなプラクティスを採用している
  - test.md # 何をテストしているかなど
  - deploy.md # Herokuのどんな機能でデプロイしているか
  
  
```

自分がこれだけの知識を持っていて、それを実装に落とし込んでいることを面接官にアピールすることが大事

どこまで書くべきかの境界は難しいが、かけるだけ書いた方が面接官へのアピールになる

- アプリケーション（Rails）とインフラストラクチャ（AWS）を作り、アプリをデプロイ（Heroku）するまで流れ
    - 要件定義 → アーキテクチャ設計 → 実装 → テスト → デプロイ

- フェーズごとに書くべきことを整理してみると…
    - 要件定義
        - ビジネス側の人から作って欲しいアプリやどのようなビジネスを実現したいかを聞く
        - スクールでは、自分自身がアプリを企画段階から作ったはずだが、実際の実務では企画するのはビジネス側
        - これは、スクールの決まりで、今すでにREADMEに書いてある
    - アーキテクチャ設計
        - アプリとインフラを含む全体のアーキテクチャ図が必要

        - アプリ側で細かく設計を語ると、MVCの話になる
            - アプリを詳細にみたアーキテクチャ図が必要
            - MVC
                - MVCのそれぞれはどのような役割があり、どのようなロジックを置くべきか決まっている。哲也のアプリケーションでは、いかにMVCに沿った実装になっているかをアピールする
                - MVCが何であるかを説明してもいいかもしれない。
                - MVCの役割（ちゃんと調べてもっと詳しく書く）
                    - M：バックエンドであり、ドメイン（ライブの概念）のルール（ライブをソフトウェア化するためにコードに落とし込むべきこと）に関するロジックを書くところ
                    - バリデーションも含む
                    - V：フロントエンドであり、画面を定義するところ
                    - C：バックエンドへのデータ入力と、フロントエンドへのデータ出力を担う
        - インフラ側で細かく設計を語ると、AWSで使っているS3の詳細な機能などの話になる
            - インフラを詳細にみたアーキテクチャ図はなくてもいい。全体のアーキテクチャ図がインフラのアーキテクチャ図を兼ねるため。
            - AWS
    - 実装
        - リーダブルコード本に書いてあるような書き方を採用していることをアピールする
        - 早期リターンを使ってますよ！などをアピールしたり
    - テスト
        - どういうことをテストしているのか（テストの設計）
        - なぜそのテストを実施しているのか
    - デプロイ
        - Herokuのどのような機能を使ってデプロイしているのか（デプロイの設計）

やるべきこと３つ

優先順位順

1. 上に書いたドキュメントを充実させる作業（GitHubで管理するやつ）







2. アプリケーションでRailsをフル活用して新しい機能の追加
3. 今すでにある実装をMVCに沿った実装に書き直す（リファクタリングという）

3は、1でMVCをドキュメントにする時にも関係するため、優先順位が高そうに見えるが、そうでもない。

面接官はアプリの実装の細かいところまで見ないため、リファクタリングしても哲也が知識を持っているアピールにならない（面接官がコードの良し悪しをちゃんと見ないため気付けない）かもしれない。

1.が一番アピールになる。
