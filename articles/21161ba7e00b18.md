# 問題
チャット機能で、初回接続時はリアルタイム更新が正常に動くが、ブラウザをリロードすると、それ以降メッセージがリアルタイムで届かなくなる(再度リロードすると直ることもあり、動作が不安定)。
# 原因
PumaがHeroku上で複数のワーカープロセス(4つ)を起動しており、ActionCableのasyncアダプターと相性が悪かったことが根本原因でした。
 （dynoは1つですが、その中でPumaが4つのワーカープロセス(PID 7, 11, 16, 26)を起動していた。）

・`config/puma.rb`の設定により、本番環境では`Concurrent.physical_processor_count`の値(このケースでは4)に応じてPumaが4つのワーカープロセスをフォークしていた

・`config/cable.yml`のActionCableアダプターが`async`になっていた
・asyncアダプターは同一プロセス内でしかメッセージを共有できない仕組み
・ブラウザがWebSocket接続するたびに、4つのワーカーのどれかにランダムに割り振られる
・メッセージ送信時の処理も別のワーカーで実行されることがあり、送信したワーカーと受信側が接続しているワーカーが異なると、broadcastが届かない
・リロードのたびに再接続先ワーカーが変わるため、症状に再現性がなく「動いたり動かなかったり」していた

# 切り分けの流れ
・ブラウザの開発者ツール(Network → WSタブ)で、WebSocket接続自体は毎回正常(101)に確立されていることを確認
・Messagesタブで、サーバーから正しくbroadcastのデータが送られてきていることを確認 → サーバー側の送信処理自体は正常と判明
・それでも画面に反映されないことがある → 受信側の接続先(プロセス)がバラバラという仮説が浮上
・heroku psで確認 → dynoは1つだけ
・heroku logs --tailで確認 → PumaがWorkers: 4のクラスターモードで動いていたことが判明(ここが悪いと発見)

# 解決策
・Heroku側で環境変数を設定し、Pumaのワーカー数を1に固定
```
bashheroku config:set WEB_CONCURRENCY=1
heroku ps:restart
```
・これにより全てのWebSocket接続が単一プロセスに集約され、asyncアダプターでも正常にbroadcastが全クライアントに届くようになった。
