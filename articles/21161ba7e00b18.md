### 開発環境
- macOS
- VSCode
- Rails 7.1.3.3
- ruby-3.2.3
- PostgreSQL 16.2
- Heroku

# 問題
・チャット機能で、初回接続時はリアルタイム更新が正常に動くが、ブラウザをリロードすると、それ以降メッセージがリアルタイムで届かなくなる。(再度リロードすると直ることもあり、動作が不安定)

# 原因
・`Puma`がHeroku上で複数のワーカープロセス(4つ)を起動しており、`ActionCable`の`asyncアダプター`と相性が悪かったことが根本原因だった。
 （dynoは1つですが、その中でPumaが4つのワーカープロセス(PID 7, 11, 16, 26)を起動していた。）
・`config/puma.rb`の設定により、本番環境では`Concurrent.physical_processor_count`の値(このケースでは4)に応じてPumaが4つのワーカープロセスをフォークしていた。
・`config/cable.yml`のActionCableアダプターが`async`になっていた
・`asyncアダプター`は同一プロセス内でしかメッセージを共有できない仕組み
・ブラウザが`WebSocket`接続するたびに、4つのワーカーのどれかにランダムに割り振られる
・メッセージ送信時の処理も別のワーカーで実行されることがあり、送信したワーカーと受信側が接続しているワーカーが異なると、`ブロードキャスト`が届かない
・リロードのたびに再接続先ワーカーが変わるため、症状に再現性がなく「動いたり動かなかったり」していた


::::details  Pumaとは？
・ユーザー と Railsアプリの間に入り、「リクエストの橋渡し」をするのが役割。
・Heroku上でワーカー数を「4」にすると、Pumaはアプリを4つの完全に独立したプロセス（部屋）に分裂させて一斉に起動する。
・アクセスしてきたユーザーは、4つの部屋のどこかにランダムに振り分けられる。
::::

::::details asyncアダプター
・ActionCableがメッセージを届けるための「配送係」の一つ
・軽量だが、「同じプロセスの中だけでしかメッセージを届けられない」 
::::






# 切り分けの流れ
・ブラウザの開発者ツールで、WebSocket接続自体は毎回正常(101)に確立されていることを確認
・`Messages`タブで、サーバーから正しく`ブロードキャスト`のデータが送られてきていることを確認 → サーバー側の送信処理自体は正常と判明
・それでも画面に反映されないことがある → 受信側の接続先(プロセス)がバラバラという可能性が浮上
・`heroku ps`で確認 → `dyno`は1つだけ
・`heroku logs --tail`で確認 → `PumaがWorkers: 4`のクラスターモードで動いていたことが判明(ここが原因だと判明した)

# 解決策
・Heroku側で環境変数を設定し、Pumaのワーカー数を1に固定
```
heroku config:set WEB_CONCURRENCY=1

Setting WEB_CONCURRENCY and restarting ⬢ live-c-app... done, v210
WEB_CONCURRENCY: 1

・Heroku上で動いているアプリのコンテナ（Dyno）を再起動する
```

heroku ps:restart

```
・これにより全てのWebSocket接続が1つのプロセスに集約され、asyncアダプターでも正常にブロードキャストが全クライアントに届くようになった。
（初回接続時からブラウザの更新後も正常にメッセージがリアルタイムで表示されるようになった）
